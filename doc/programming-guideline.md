# Руководство по программированию

## 1. О документе
Это руководство описывает правила написания кода на языке препроцессора PlantUML для проекта PlantUML DSL.

### 1.1. Цель
Установить единые стандарты написания переиспользуемых функций/процедур и сопроводительного кода в проекте.

### 1.2. Термины и определения
- **DSL** - Domain-Specific Language - предметно-ориентированный язык, в данном случае надстройка над PlantUML.
- **Препроцессор** - Механизм PlantUML для макроподстановок и выполнения логики до рендеринга диаграммы.
- **Built-in function** - Встроенная функция PlantUML с префиксом `%` (например, `%load_json`, `%size`).
- **Guard clause** - Ранний выход из функции при невыполнении входных условий (валидации).
- **DocString** - doc-комментарий перед функцией/процедурой с описанием сигнатуры: назначение, аргументы, возвращаемое значение, пример вызова.
- **Early return** - ранний выход из функции при невыполнении входных условий, без выполнения остального кода.
- **Fail-fast** - принцип немедленного прерывания выполнения при обнаружении ошибки, обычно через `!assert`.
- **Побочный эффект (side-effect)** - неявное изменение состояния программы (глобальных переменных, диаграммы), выходящее за рамки возвращаемого значения.
- **Идемпотентность** - свойство функции давать одинаковый результат при повторном вызове с одними и теми же аргументами.
- **DRY** (Don't Repeat Yourself) - принцип избегания дублирования кода через выделение общей логики в переиспользуемые функции/процедуры.
- **Arrange / Act / Assert** - трёхфазный паттерн тестирования: подготовка данных, выполнение действия, проверка результата.

### 1.3. Ссылки

## 2. Структура и стиль
### 2.1. Структура проекта
Код проекта организован следующим образом:
- `src/` - исходный код проекта,
- `tests/` - автотесты для кода,
- `playground/` - различные примеры использования и "песочница".

В каталоге `src/` находятся подкаталоги отдельных функциональных модулей, содержащих DSL и вспомогательные инструменты:
- `C4/` - архитектурная модель C4,
- `Data/` - концептуальная/логическая модель данных,

- ... - другие аналогичные модули.

- `Utils.puml` -

Каталог `tests/` полностью повторяет структуру `src/`.  
Каталог `playground/` лишь частично повторяет структуру `src/`, потому что содержит еще подкаталоги для различных кейсов.

### 2.3. Стиль оформления кода 
- Кодовые файлы PlantUml имеют расширение `.puml`.
- Кодовый файл начинается со стандартной директивы `@startuml` и заканчивается директивой `@enduml`.
- Сразу после заголовка идет секция `!include`.
- После идет описание файла в виде блока комментариев с разделителями.
- Кодовый файл содержит исходный код функций (`!function`) и процедур (`!procedure`) на языке препроцессора PlantUml.
- Функция/процедура начинается заголовком `DocString`, описывающим сигнатуру (входные/выходные параметры, ограничения и т.д.).
- Функции/процедуры отделены друг от друга разделителями секций: `''---------------------` для лучшего структурирования.
- Логические блоки кода разделены 1 или более пустой строкой.
- Максимальная длина строки - 120 символов.

Пример структуры кодового файла:
```plantuml
@startuml
!include ../src/Utils.puml
''----------------------------------------------------------------------------------------------------------------------
'' Модуль работы с коллекциями JSON.
'' Зависимости: Utils.puml
''----------------------------------------------------------------------------------------------------------------------
'' Get first element of collection.
!function $get_first_item($array)
    '' ...
!endfunction
''----------------------------------------------------------------------------------------------------------------------
@enduml
```

## 3. Комментарии
- Комментарии в коде начинаются с двойного апострофа: `''`. 
- **DocString** размещается непосредственно перед функцией/процедурой и содержит:
  - Краткое описание функции/процедуры.
  - Описание аргументов и возвращаемого значения (для функций) в секции `args:` в виде: `'' $num - номер позиции элемента'` (точка не ставится).
  - Пример вызова в секции `example:` в виде `'' !$result = $function_name($arg)`.
- **inline-комментарии** (`'' some text`) используются для пояснения нетривиальной логики внутри тела функции.
- **разделители** (`''--------`) служат только для визуального разделения логических секций файла, не для пояснений.
- Запрещено оставлять закомментированный код в основной ветке.

Пример использования комментариев:
```plantuml
''----------------------------------------------------------------------------------------------------------------------
'' Загружает JSON из указанного пути файла.
'' args: $path - полный путь к файлу
'' example: !$data = $load_json_safe("config.json")
''----------------------------------------------------------------------------------------------------------------------
!function $load_json_safe($path)
    !if %file_exists($path)
        !return %load_json($path)
    !else
        !return {}  '' возвращаем пустой объект, если файл не найден
    !endif
!endfunction
```

## 4. Именование
Выбор имён влияет на читаемость и понятность кода. Следуйте этим соглашениям:

### 4.1. Модуль
Имя модуля в PascalCase, например: `Data`.

### 4.2. Файл
Имя кодового файла в PascalCase, например: `Utils.puml`.

### 4.3. Функция или процедура
Правила именования отличаются в зависимости от назначения и области видимости.

#### 4.3.1. Открытая функция, представляющая DSL
Открытая функция представляет открытый (публичный) API "только для чтения" для пользователей DSL.
Имя открытой функции в PascalCase и без префиксов, например:
- `Diagram(...)` - объявляет диаграмму с определенными настройками/опциями.
- `Persons(...)` - подключает к диаграмме зарегистрированных ранее Акторов.
- `Relations(...)` - подключает к диаграмме зарегистрированные ранее отношения между элементами.

#### 4.3.2. Закрытая функция, представляющая DSL
Закрытая функция представляет закрытый API "регистрации элементов" для пользователей DSL.
Имя закрытой функции в PascalCase и c префиксом `$`, например:
- `$Person(...)` - регистрирует Актора с заданными атрибутами.
- `$Dependency(...)` - регистрирует отношение типа "Зависимость" между двумя элементами.

#### 4.3.3. Закрытая вспомогательная функция
Закрытые вспомогательные функции - это аналог build-in функций, только реализованных в данном проекте.
Имя функции в SnakeCase и с префиксом `$`, например:
- `$build_element_metadata(...)` - создание JSON-объекта с атрибутами элемента.
- `$append_to_rendering_queue(...)` - добавление дескриптора элемента к очереди рендеринга.

#### 4.3.4. Тестовая процедура
Тестовые процедуры, очевидно, проверяют различные тестовые утверждения.
Имя процедуры в SnakeCase, не имеет префикса и соответствует паттерну `<unit>_<condition>_<expected>`, например:
-  `get_library_version_returns_semantic_version_number()` - проверяет, 
что функция `get_library_version()` возвращает семантический номер версии библиотеки.  

### 4.4. Переменная
Переменные начинаются с символа `$` и именуются по правилам SnakeCase в **нижнем** регистре, например:  
`$element`, `$actor_name`, `$canonical_domain_model`.

### 4.5. Константа
Константы начинаются с символа `$` и именуются по правилам SnakeCase в **верхнем** регистре, например:  
`$DEBUG`, `$SCREEN_WIDTH`.

## 5. Функции и процедуры
### 5.1. Функция
- Описывается директивой `!function`.
- Возвращает значение через `!return`.
- Должна быть **идемпотентной**: одинаковые входные аргументы приводят к одинаковым результатам.
- Не должна иметь побочных эффектов.

### 5.2. Процедура
- Описывается директивой `!procedure`.
- Не возвращает значение (нет `!return`).
- Используется для изменения состояния объектов или конструирования/модификации диаграмм.
- Может иметь побочные эффекты.

Общие правила:
- Одна функция/процедура = одна задача.
- Используйте выразительные, **говорящие** имена.
- Используйте **guard clause** в открытых функциях.
- Документируйте назначение и пример использования в `DocString` перед объявлением.

## 6. Объявление переменных
- **Локальные переменные** объявляются внутри функции/процедуры через директиву `!$name = value`, где
  - `!` - префикс директивы присвоения (statement),
  - `$name` - имя переменной (начинается с `$`),
  - `value` - присваиваемое значение.
- Область видимости локальной переменной ограничена текущим блоком (`!function`/`!procedure`).
- Избегайте глобального состояния (кроме специальных случаев). 
- JSON - основной составной тип для структурированных данных:
  `!$config = {"ver": "1.0", "settings": {"timeout": 60}}`
- Константы помечайте комментарием `'' const`, например:
  `!$MAX_RETRIES = 3  '' const`

## 7. Выражения и управляющие конструкции

PlantUML препроцессор поддерживает следующие управляющие конструкции:

### `!if` / `!else if` / `!else`
- Предпочитайте **early return** (ранний выход) вместо глубокой вложенности.
- Сравнивайте через `==`, `!=`, не через `=`.

Пример:
```plantuml
!function $validate_input($input)
    !if $input == ""
        !return %false()
    !else if %size($input) > 255
        !return %false()
    !endif
    !return %true()
!endfunction
```

### `!foreach`
- Используется для итерации по JSON-массивам.
- Подходит для обработки коллекций.

```plantuml
!foreach $item in $array
    note right: $item
!endfor
```

### `!while`
- Избегайте использования, если есть альтернатива через `!foreach` с диапазоном.

### Вызовы built-in функций
- Всегда указывайте скобки, даже если аргументов нет: `%true()`, не `%true`.

## 8. Обработка ошибок
PlantUML препроцессор не имеет механизма исключений, поэтому обработка ошибок основана на проверках.

- **Guard clauses** - проверяйте входные аргументы в начале функции через `!assert`, например:
  ```plantuml
  !function $process_data($data)
      !assert $data != ""
      !assert %size($data) > 0
      '' ... основная логика
  !endfunction
  ```
- **Fail-fast** для внутренних инвариантов - сразу проверяйте условия, нарушение которых указывает на ошибку в коде.
- Для эмуляции исключений (и безусловного прерывания выполнения) используйте процедуру `$fail()` из `Utils.puml`.
- Никогда не замалчивайте ошибки: либо `$fail()`, либо `!assert`, либо явный возврат.
- Для отладочной печати (непосредственно на диаграмму) используйте процедуру `$trace()` из `Utils.puml`.

Пример обработки:
```plantuml
!function $load_config($path)
    !if %not(%file_exists($path))
        $fail('Config file not found: "%1"', $path)
    !else
        !return %load_json($path)
    !endif
!endfunction
```

## 9. Переносимость
Правила для обеспечения работы кода на разных платформах и окружениях:

- **Только относительные пути** в `!include`:
  ```plantuml
  !include ../src/Utils.puml   '' правильно
  !include C:/project/src/Utils.puml  '' запрещено
  ```
- **Кодировка файла**: UTF-8.

## 10. Повторное использование
Принципы проектирования переиспользуемых модулей:
- **Без побочных эффектов при импорте** - модуль не должен выполнять логику на верхнем уровне (вне функций/процедур), 
это позволяет безопасно подключать его через `!include`.
- **DRY** - следует исключать дублирование кода, выделяя общие части в виде функций/процедур.
- **Проектируйте функции как "чистые"** (идемпотентные, без глобального состояния) - 
это упрощает их переиспользование в других модулях.

## 11. Тестирование

Тесты размещаются в `tests/` и запускаются через `task test`.

### 11.1. Структура тест-файла
- Один тест-файл на один исходный модуль: `Foo.puml` и `Foo.Tests.puml`.
- Тест импортирует исходный модуль через `!include ../src/<FileName>.puml`.

### 11.2. Паттерн Arrange / Act / Assert
Каждая тестовая процедура следует этому шаблону:
```plantuml
!procedure get_library_version_returns_semantic_version_number()
    '' Arrange, Act
    !$actual = $get_library_version()

    '' Assert
    !$pattern = "/regex/"
    !assert %size(%splitstr_regex($actual, $pattern)) == 1
!endprocedure
```

### 11.4. Запуск тестов
- Тестовые процедуры вызываются в секции `'' Tests RUN!` в конце файла.
- Команда `task test` обязательно запускается перед каждым коммитом автоматически (через git hooks).
- Файл с тестами должен содержать директиву `caption OK`.

Пример полного тест-файла:
```plantuml
@startuml
!include ../src/Utils.puml
''------------------------------------------------------------------
'' Tests
''------------------------------------------------------------------
!procedure get_library_version_returns_semantic_version_number()
    '' Arrange, Act
    !$actual = $get_library_version()

    '' Assert
    !$pattern = "/^v\d+\.\d+\.\d+$/gm"
    !assert %size(%splitstr_regex($actual, $pattern)) == 1
!endprocedure
''------------------------------------------------------------------
'' Tests RUN!
''------------------------------------------------------------------
get_library_version_returns_semantic_version_number()
''------------------------------------------------------------------
caption OK
@enduml
```

## 13. Проблемы интерпретации
Особенности работы препроцессора PlantUML и ограничения:

### 13.1. Запрещено
- **Рекурсивные `!include`** - PlantUML не поддерживает цикл зависимостей.
- **Side-effects в `!function`** - функции не должны выводить в диаграмму или модифицировать глобальное состояние.
- **Отсутствие `!return` в `!fucntion`** - синтаксическая ошибка.

### 13.2. Built-in функции (`%`-префикс)
- Доступность встроенных функций зависит от версии PlantUML.
- Функции для модификации JSON-объектов **недоступны** из файлов `@startuml/@enduml`, 
поэтому их приходится эмулировать операциями со строками и парой доступных функций: `%str2json()` и `%string()`.

При добавлении новой built-in функции - убедитесь, что она доступна в минимальной поддерживаемой версии PlantUML (указать в README).
